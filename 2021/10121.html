javascript 的设计模式

模块化模式

模块化模式：让内部成员不外泄到全局中，将内部信息封装成一个模块。只能通过特定的方法才可以进行访问内部成员信息。比如说购物车中数组是不能直接让外部进行访问的，只能通过其域中特定的方法才可以进行访问。

var buy = (function () {
  var buyList = []
  //增加购物车数据方法
  function addItem(values) {
    buyList.push(values)
  }
  //得到购物车数据数量的方法
  function getItem() {
    conlose.log(buyList.length)
  }

  return {
    addItem: addItem,
    getItem: getItem
  }
})()

buy.addItem({
  name: 'one',
  price: 1
})

buy.addItem({
  name: 'two',
  price: 2
})

buy.getItem() //2

console.log(buy.buyList) //这个是不可以直接访问的,直接访问的结果是undefined

单例模式

单例模式是一种十分常用但却相对而言比较简单的设计模式。它是指在一个类只能有一个实例，即使多次实例化该类，也只返回第一次实例化后的实例对象。单例模式不仅能减少不必要的内存开销, 并且在减少全局的函数和变量冲突也具有重要的意义。
使用闭包实现单例模式

let Create = function () {
  let instance = null
  return function (name) {
    this.name = name
    if (instance) {
      return instance
    }
    return (instance = this)
  }
}

Create.prototype.getName = function () {
  console.log(this.name)
}

let xiaoming = new Create('xiaoming')
// console.log(xiaoming.getName());
xiaoming.getName() //xiaoming
let xiaoliu = new Create('xiaoliu')
// console.log(xiaoliu.getName());
xiaoliu.getName() //xiaoming

利用一个变量 instance 判断是否已经创建实例，如果之前没有创建实例就将 this 传给 instance。当第二次创建实例的时候，虽然向构造函数中传入参数了但是因为 instance 已经存在 所以返回的还是上一次创建的实例。

寄生式组合继承

寄生式组合继承相对来说是最为好的一种继承方式

//让子类原型继承父类原型
function inheritObject(o){
  //创建一个过渡函数对象
  function F(){}
  //让F的原型对象继承父类原型对象
  F.prototype=o
  //返回F实例，这个实例继承了父对象的原型
  retun new F()
}

//寄生式继承
function inheritPrototype(subClass,superClass){
  //创建父类原型副本，存储在变量中
  var p=inheritObject(superClass.prototype)
  //修正因为重写子类原型导致子类的 constructor属性被修改
  p.constructor = subClass
  //子类原型继承父类原型
  subClass.prototype=p
}

//构建父类构造对象
function SuperClass(name){
  this.name=name
  this.colors=['red','black']
}
//父类原型对象方法
SuperClass.prototype.getName=function(){
  console.log(this.name)
}
//构建子类构造对象
function SubClass(name,time){
  //继承父类构造
  superClass.call(this,name)
  this.time=time
}
//子类原型对象方法
SubClass.prototype.getTime(){
  console.log(this.time)
}

//创建子类实例对象
var instance1=new SubClass('app book',2020)
var instance2=new SubClass('css book',2021)

instance1.colors.push('blue')

console.log(instance1.colors)//['red','black','blue']
console.log(instance2.colors)//['red','black']

instance2.getName()//css book
instance2.getTime()//2021

一开始我对 p.constructor = subClass 这句话有歧义，因为我感觉去掉也是可以的。但是我上网查询了一下资料。不把 constructor 指回子类是可以的，但是后面添加原型方法就会出现问题。constructor 一般在闭包不能直接访问的情况下可以通过 constructor 进行添加一些方法。

简单工厂模式

当我们实现登录功能的时候，我们可能会用到警示框，提示框各种有些区别的框框。我们第一个想到的是将其封装成一个一个方法。但是只能一味的封装类，可能每一个框中有一些东西出现了差异之后，这时我们就需要再创建一个类。这实在是太麻烦了

//自定义警示框 不使用工厂类创建
//需求：需要一个警示框，错误提示框，确认框(确认框中需要添加一个注册按钮)
var LoginAlert = function (text) {
  this.content = text
}
LoginAlert.prototype.show = function () {
  //显示警示框
  console.log('显示警示框')
}
//警示框

//需要创建一个新类
var LoginComfim = function (text) {
  this.content = text
}
LoginComfim.prototype.show = function () {
  //展示确认框
}
//当内容又有一点差异时

var LoginPromat = function (text) {
  this.content = text
}
LoginPromat.prototype.show = function () {
  //展示
}

//使用简单工厂模式

var PopFactory = function (name) {
  switch (name) {
    case 'alert':
      return LoginAlert
    case 'confim':
      return LoginComfim
    case 'prompt':
      return LoginPromat
  }
}

var Factory = new PopFactory('alert')
var alert = new Factory('警示框')
console.log(alert.content) //警示框
alert.show() //显示警示框

当我们使用了这个工厂之后，再注册模块中也可以使用在完成登录模块时封装的方法。而且我们只需要记住这个工厂就好。
但是我们发现这封装的三个类中有很多相似点，所以我们遇到这种情况需要使用另一种工厂模式

//另一种简单工厂化模式
function createPop(type, text) {
  //创建一个对象，并对对象拓展属性和方法
  var o = new Object()
  o.content = text
  o.show = function () {
    //共有方法
  }
  if (type == 'alert') {
    //警示框差异部分
    console.log('2警示框')
  }
  if (type == 'prompt') {
    //提示框差异部分
    console.log('提示框')
  }
  if (type == 'confim') {
    console.log('确认框')
  }
  return o
}

var userNameAlert = createPop('alert', '另一种工厂警示框') //2警示框
console.log(userNameAlert.content) //另一种工厂警示框

安全类，安全工厂方法

我们在使用类的时候如果不进行实例的话，可能就会出现问题。小白可能在实例化的时候忘记 new 关键字。这个时候就可能出现一些问题。所以我们需要使用安全类

//不使用安全模式的类
var Demo = function () {}
Demo.prototype = {
  show: function () {
    console.log('成功获取')
  }
}
var demo = new Demo()
demo.show() //成功获取

var demo2 = Demo
demo2.show() //报错
//安全模式类
var Test = function () {
  if (!(this instanceof Test)) {
    return new Test()
  }
}
Test.prototype = {
  show: function () {
    console.log('安全获取')
  }
}
var d = Test()
d.show() //安全获取

我们上面说了怎样使用简单工厂模式，但是当我们需求过多的时候。我们不仅仅要添加类，还要修改工厂函数。这简直更加麻烦。所以我们这时候可以使用安全工厂方法

var Factory = function (type, count) {
  if (this instanceof Factory) {
    //这时候使用者new了实例,返回对应的方法
    let s = new this[type](count)
    return s
  } else {
    //这时使用者没有new，应该构造一个工厂实例然后返回
    return new Factory(type, count)
  }
}
//添加工厂方法
Factory.prototype = {
  java: function (count) {
    consloe.log(count)
  },
  javascript: function (count) {
    consloe.log(count)
  }
}
var use = Factory('java', '使用java') // 使用java
//我们使用的时候 无论使不使用new 都可以使用工厂中的方法

当我们没有使用 new 的时候 这时候的 this 指向的 window 全局对象，所以 this instanceof Factory 这个就时 false 然后代码执行 else{} 返回了一个实例的 Factory 然后这时候的 this 指向的就是 Factory 了。然后 this 其实就是这个工厂构造，然后里面有这几个工厂方法.this[type]=this.java。 所以返回的 s 其实就是 java 这个方法了。

抽象工厂模式

抽象工厂更加体现类的继承关系。一般是先创建抽象工厂，然后创建具体抽象工厂，最后是产品。

//创建抽象工厂类
var AbstractFactory = function (subType, superType) {
  //判断抽象工厂中是否有此具体抽象工厂
  if (typeof AbstractFactory[superType] === 'function') {
    //创建缓存类
    var F = function () {}
    //继承抽象工厂类的属性
    F.prototype = new AbstractFactory[superType]()
    //指回
    subType.constructor = subType
    //子类继承父类
    subType.prototype = new F()
  } else {
    return '没有此抽象工厂'
  }
}
//创建具体抽象工厂
AbstractFactory.Car = function () {
  this.type = 'car'
}
//创建抽象产品类
var BWM = function (price, speed) {
  this.price = price
  this.speed = speed
}
//继承父类(具体抽象工厂)
AbstractFactory(BWM, 'Car')
//抽象产品类的方法
BWM.prototype.getPrice = function () {
  return this.price
}
BWM.prototype.getSpeed = function () {
  return this.speed
}

//创建具体产品
var threeXi = new BWM('价格', '速度')
conlog.log(threeXi.getPrice()) //价格
conlog.log(threeXi.getSpeed()) //速度
